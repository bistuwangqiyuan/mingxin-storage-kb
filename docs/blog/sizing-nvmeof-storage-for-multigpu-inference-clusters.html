Sizing NVMe‑oF Storage for Multi‑GPU Inference Clusters

Published 2026-07-20 · Mingxin Technology Insights

Multi‑GPU inference clusters move large language models (LLMs) and other transformer workloads off single‑host memory and into distributed memory + storage ecosystems. Sizing NVMe‑over‑Fabric (NVMe‑oF) for such clusters is primarily about meeting latency SLOs and aggregate throughput while keeping tail latencies predictable under multi‑tenant concurrency.

Key inputs: what you must measure first

Before you do any capacity math, gather measured or well‑validated estimates for:

  • Model characteristics: model size (parameters), context length, whether KV cache is used, and per‑token activation memory.
  • Inference SLOs: tokens/sec per GPU, max latency and TTFT (time‑to‑first‑token), P99/P99.9 targets.
  • Concurrency and batching: number of concurrent streams per GPU, batch sizes, and request arrival patterns.
  • KV cache behavior: working set size, average KV entry size, and expected cache hit ratio under your workload.
  • IO profile: average read size (bytes), reads per token, sequential vs random access, and write patterns (checkpointing, logs).

These inputs let you convert model behavior into storage I/O demand.

Converting workload to IOPS and bandwidth

Two outputs matter for storage sizing: IOPS (random IO operations/sec) and throughput (GB/s). Use these formulas:

  • Required IOPS = tokens_per_sec_total * reads_per_token * (1 − cache_hit_rate)
  • Required bandwidth (GB/s) = Required IOPS * average_read_size_bytes / 1e9

Example calculation (illustrative):

  • tokens_per_sec_total = 20,000 tokens/sec across cluster
  • reads_per_token = 1 KV lookup per generated token
  • cache_hit_rate = 0.7 (70% hits in NVMe cache)
  • average_read_size = 32 KB

Then Required IOPS = 20,000 * 1 * 0.3 = 6,000 IOPS.
Required bandwidth = 6,000 * 32 KB ≈ 192 MB/s (0.19 GB/s).

Note: this example shows that even modest IOPS can require only sub‑GB/s bandwidth if cache hit ratio is high. But tail latency and concurrency place heavier demands on low latency IOPS rather than raw GB/s.

Latency and tail behavior: why NVMe‑oF choice matters

  • Tail latency (P99/P99.9) drives user perceived latency and TTFT. NVMe‑oF with RDMA (RoCE/IB) typically offers lower and more predictable latency than TCP‑based NVMe‑oF under similar conditions, but requires network QoS (ECN/DSCP) and careful switch configuration.
  • Network contention, QoS misconfiguration, and host CPU scheduling can cause long tails. Budget headroom: plan for <P99 SLO / 2 latency per storage hop during sizing and validate with synthetic load tests.

Network and host‑side sizing

  • NIC bandwidth: compute GB/s requirement across hosts. Include replication or mirroring traffic if using synchronous writes. Example: a 100 GbE link provides ~12.5 GB/s raw; allow 30–40% headroom for other traffic and RoCE overheads.
  • CPU & PCIe: NVMe‑oF client stacks (kernel or SPDK) consume CPU cycles; factor in CPU cores per host to maintain queue depth and low latency.
  • Queue depth and IO parallelism: GPUs generate many concurrent outstanding IOs; ensure NVMe namespaces and controllers can absorb the queue depth without head‑of‑line blocking.

Tiering and cache strategies

  • In‑GPU memory is fastest but small. Use an in‑host RAM cache (KV cache) backed by remote NVMe‑oF for the larger working set.
  • KV cache tiering reduces read miss volume to NVMe by maintaining a high hit ratio in RAM and local NVMe, but sizing must account for prefetch efficacy and eviction churn.
  • Prefetching can smooth IO but may increase bandwidth; balance prefetch aggressiveness against network bandwidth limits.

Reliability, capacity and cost tradeoffs

  • Replication and erasure coding add network and IO overhead — size for effective bandwidth after replication (e.g., a 2x mirror doubles bandwidth demand for writes).
  • Use QoS features (IOPS and bandwidth limits) at the storage target to protect tail latency in multi‑tenant clusters.

Vendor evaluation checklist (storage platform matters)

When comparing NVMe‑oF platforms, evaluate:

  • Latency and tail latency under realistic, concurrent workloads (P95/P99/P99.9).
  • Measured IOPS and throughput for your read sizes (4K–64K).
  • Headroom for concurrency and mixed workloads.
  • Network protocol support (RoCE v2, IB, TCP) and guidance for switch/QoS configuration.
  • Integration with KV cache tiering and prefetch controls.
  • Reproducible benchmark reports and joint validation options (gate‑based acceptance).

Comparison table: storage options for inference clusters

Option Typical latency profile Scalability Best use case Tradeoffs
Local NVMe (per‑host) Lowest tail latency Limited to host capacity Small clusters, when model fits host memory Not shareable across hosts, expensive to scale
NVMe‑oF (RDMA) Low latency, predictable tail High (shared pool) Distributed KV caches, multi‑GPU clusters Requires QoS & network tuning, RDMA ops expertise

SATA/NVMe over FC or object store High latency Very high capacity Cold storage, checkpoints Not suitable for low latency inference

Practical validation and test plan

  1. Run an isolated microbenchmark that reproduces KV cache miss pattern at target concurrency. Measure P50/P95/P99 latency.
  2. Add background traffic and multi‑tenant load and re‑measure tails.
  3. Validate end‑to‑end: model TTFT and tokens/sec for target concurrency.
  4. Tune queue depth, NIC offloads, and storage QoS iteratively.

Key takeaways

  • Start from tokens/sec, reads_per_token, average_read_size and cache_hit_rate to compute IOPS and GB/s needs.
  • Tail latency (P99/P99.9) is often the limiting factor — design for headroom and enforce QoS.
  • NVMe‑oF (RDMA) gives better tail behavior but needs network expertise; TCP NVMe‑oF is simpler but can be less predictable under contention.
  • Validate with signed, reproducible benchmarks and run joint acceptance tests before production.

  • Tune QoS, queue depths, and caching; repeat until SLOs met.

Sizing NVMe‑oF for multi‑GPU inference is less about raw GB/s and more about predictable low tail latency under the real concurrency pattern of your application. Use measurement, conservative headroom, and vendor‑validated benchmarks as the basis for procurement and operational tuning.
