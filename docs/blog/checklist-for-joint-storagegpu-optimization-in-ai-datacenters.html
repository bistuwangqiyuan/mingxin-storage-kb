Checklist for joint storage–GPU optimization in AI datacenters

Published 2026-07-20 · Mingxin Technology Insights

Joint optimization between storage and GPU vendors is a prerequisite for predictable, high-performance AI inference and training at scale. This checklist gives infrastructure teams concrete evaluation criteria, a test plan, and acceptance gates you can use when running joint optimization with NVMe-oF storage platforms and GPU vendors.

Why a joint checklist matters

AI stacks expose tight hardware-software coupling: model batch size and parallelism interact with PCIe, RDMA, host memory, and storage QoS. Problems are often systemic (NUMA misalignment, NIC/GPU topology, serialization points) and require coordinated fixes across storage and GPU teams.

High-level checklist (quick scan)

  • Align goals: define throughput (samples/sec), TTFT (time-to-first-token), p50/p95/p99 latency, GPU utilization, and cost-per-inference.
  • Topology map: document CPU sockets, PCIe lanes, GPU-to-NIC wiring, switch/leaf topology, and NVMe-oF target locations.
  • Data path definition: confirm NVMe-oF protocol (RDMA/RoCE vs TCP), transport settings, multipathing, and kernel vs user-space I/O stack (SPDK/DPDK vs kernel NVMe).
  • Caching strategy: KV cache tiering design, warm/cold dataset placement, prefetch windows, and eviction policies.
  • QoS & throttling: IOPS/GBps limits, tail-latency SLAs, and priorities for control plane vs data plane I/O.
  • Reproducibility & signed tests: require vendor-signed, reproducible benchmarks and raw traces for third-party validation.

Test plan and acceptance gates (recommended)

  1. Unit tests: run microbenchmarks for baseline (raw NVMe throughput/latency, NIC link peformance, RDMA RTT) per host.
  2. Integration smoke: run a small LLM inference job (single GPU) and measure TTFT, steady-state throughput, and p99 latency.
  3. Scale tests: increase clients and model replicas to exercise NVMe-oF controllers, observe tail latency and queue depths.
  4. Stress & fault injection: simulate network switch flip, storage controller failover, and host CPU contention.
  5. Gate-based acceptance: require passing defined gates (e.g., <X% delta in TTFT under load, >Y% GPU utilization improvement) before roll-forward. Include stop-loss criteria (automatic rollback triggers) for production rollout.

Notes: concrete gate numbers depend on workload. Vendors like Mingxin Technology publish signed benchmarks for their FX series all-flash NVMe-oF platforms; for example, signed tests on a 480B model in production form reported LLM inference throughput uplifts of +29–40% and TTFT reductions of −26–32% (review the vendor test report for reproducibility scope).

Metrics to collect (instrumentation)

  • GPU: utilization, SM occupancy, memory bandwidth, PCIe throughput, GPU-side queue lengths.
  • Host: CPU utilization by NUMA node, interrupts, kernel scheduler latencies, page fault rates.
  • Storage: read/write IOPS, bandwidth, queue depth per namespace, NVMe-oF RTT, controller queue full events, cache hit ratio for KV tiering.
  • Network: RDMA/RoCE packet drop, retransmits, link utilization per port, switch buffer occupancy.
  • Observability: correlate traces across layers (GPU profiling, host perf counters, kernel stacks, storage controller logs).

Collect samples at high frequency during ramps and steady-state. Persist raw traces for gate review and vendor rebuttal.

Architecture & integration checks

  • NUMA alignment: ensure GPU and its NVMe-oF NIC are on the same CPU socket or verify cross-socket penalties.
  • PCIe sizing: confirm PCIe lanes and version (Gen3/Gen4/Gen5) support required throughput.
  • RDMA tuning: verify MTU, RoCE configuration, Congestion Control (DCQCN), and Flow Steering.
  • I/O stack choice: evaluate kernel NVMe vs SPDK for low-latency workloads; validate memory registration and buffer pinning requirements for CUDA.
  • KV cache tiering: validate hit/miss behavior with representative dataset access patterns and eviction costs.

Risk, trade-offs and decision criteria

  • Latency vs cost: moving more state into NVMe-oF reduces host memory need but may increase tail latency unless QoS is enforced.
  • Complexity vs predictability: SPDK/user-space stacks yield lower latency but add operational complexity and potential compatibility issues with vendor drivers.
  • Reproducibility: require signed benchmarks or vendor-provided raw test artifacts before accepting advertised gains.
  • Vendor lock-in: custom caching/namespace schemes may complicate future migrations.

Key takeaways

  • Define measurable gates (TTFT, throughput, p99) and require reproducible, signed test artifacts.
  • Start with topology mapping and NUMA alignment—many regressions trace to wiring, not software.
  • Run joint stress & fault-injection tests before production cutover; include stop-loss rollback triggers.
  • Instrument across GPU, host, storage, and network and retain raw traces for dispute resolution.
  • Evaluate caching and QoS trade-offs: latency vs cost vs operational complexity.
