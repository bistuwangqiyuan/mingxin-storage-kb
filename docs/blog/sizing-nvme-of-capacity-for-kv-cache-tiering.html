Sizing NVMe-oF Capacity for KV Cache Tiering

Published 2026-07-20 · Mingxin Technology Insights

Overview

Sizing NVMe-oF capacity for a KV (key-value) cache tier is an engineering exercise that combines access-pattern profiling, capacity math, IOPS/throughput planning, flash endurance estimation, and operational headroom. This note gives concrete metrics to collect, step-by-step sizing formulas, a worked example, and operational checks you must include before procurement or deployment.

Key metrics you must measure first

  • Request rate (R): total requests/sec to the KV layer.
  • Access distribution: frequency rank per key (real histogram or Zipf parameter). This determines how much of the request stream is covered by a small subset of keys.
  • Working set (WS): total size of unique keys+values that your application touches over the hot window (GB/TB).
  • Average object size (S_avg): mean bytes per value (and median/percentiles).
  • Latency target (P99/P95): acceptable end-to-end latency for hits and misses.
  • Hit-rate target (H): fraction of requests you want served by the NVMe-oF cache.
  • Write/mutation rate to keys and write amplification (WA) from your caching strategy (write-through vs. write-back vs. read-only).
  • Growth & headroom: expected growth rate and peak multiplier for burst handling (commonly 1.2–2.0x).

Collect these from production traces or a representative load generator; sampling at per-key level is essential when access is skewed.

Basic sizing formulas and approach

  1. Object coverage method (recommended if you have a frequency histogram):

    • Sort keys by request frequency. Compute cumulative request coverage and cumulative bytes stored.
    • Find the smallest cumulative bytes (C_bytes) such that cumulative request coverage >= H.
    • Required cache capacity C = C_bytes * (1 + metadata_overhead + slack).

    This method captures non-uniform access patterns; it is the most accurate for real-world KV workloads.

  2. Uniform-access approximation (fast but conservative):

    • If accesses are uniform or you lack a histogram, use: C = WS * H
    • Add 10–30% for metadata and fragmentation: C_final = C * (1 + 0.1..0.3)

  3. IOPS and throughput for backend sizing (miss handling):

    • Miss rate M = 1 - H
    • Miss IOPS = R * M
    • Miss throughput (GB/s) = R * M * S_avg

  4. Endurance and write planning:

    • Write bytes/sec to flash = writes/sec * avg_write_size * WA
    • TB/day = (write_bytes/sec * 86,400) / 1e12
    • DWPD (writes per day relative to device capacity) = TB/day / device_capacity_TB

    Compare DWPD to vendor endurance targets (or design for conservative 1–5 DWPD depending on drive class).

  5. Parallelism and latency headroom:

    • NVMe-oF performance scales with connection count, queue depth, and controllers. Plan for multiple namespaces and parallel clients to avoid serialization.
    • Target device utilization < 60–70% for latency-sensitive hot-cache workloads.

Worked example (illustrative)

Assumptions from traces:

  • R = 200,000 requests/sec
  • WS = 10 TB of unique key-values
  • S_avg = 16 KB
  • Observed skew: top 15% of keys account for 80% of requests
  • Hit-rate target H = 0.92 (92% of requests served by cache)
  • Metadata + slack = 15%

Step A — Capacity from coverage:

  • If top 15% of keys = 80% coverage, to reach 92% you might need top ~30% of keys. If top 30% of WS equals 3 TB (from the cumulative bytes curve), then C_bytes ≈ 3 TB.
  • Add overhead: C = 3 TB * 1.15 = 3.45 TB -> round up to 4 TB per cache node for headroom.

Step B — Backend IOPS/throughput (misses):

  • Miss rate M = 0.08; Miss IOPS = 200k * 0.08 = 16,000 ops/sec
  • Miss throughput = 200k * 0.08 * 16 KB ≈ 256 MB/s (sustained)

Step C — Endurance rough check (if cache writes on fill only):

  • If fills equal misses and each miss causes a write of 16 KB, write bytes/sec ≈ 16,000 * 16 KB ≈ 256 MB/s
  • TB/day ≈ 256 MB/s * 86,400 / 1e12 ≈ 22 TB/day
  • If a 4 TB device is used per node, DWPD ≈ 22 TB/day / 4 TB ≈ 5.5 DWPD — choose higher-endurance enterprise NVMe or reduce write amplification via caching policy or host-side buffering.

These numbers illustrate why distribution-based coverage is critical: uniform math would have required C = WS * H = 10 TB * 0.92 = 9.2 TB (much larger) and overly pessimistic.

Operational considerations

  • Failure & replication: If you replicate data across nodes for availability, multiply capacity by replication factor. For read-only caches you might accept best-effort rebuilds; for stateful caching plan for fast rebuild/rehydration.
  • Fabric: RDMA (RoCE/iWARP) vs. TCP NVMe-oF impacts latency and CPU usage. Benchmark with your switch and host stack.
  • Metadata store: small key-value metadata can be large in aggregate — account for tens of bytes per entry.
  • Eviction policy: LRU vs frequency-based affects required capacity. Frequency-aware (LFU, TinyLFU) often reduces required capacity compared with pure LRU on skewed workloads.

Validation and testing

  • Replay production traces against a test NVMe-oF cache cluster and measure achieved hit rate for a given cache size.
  • Run endurance burn-in tests that match real write patterns (not synthetic sequential writes).
  • Gate-based acceptance: define stop-loss criteria (latency, miss amplification) and fail the test if not met.

Key takeaways

  • Always start with a per-key frequency histogram — it drives the most accurate cache sizing.
  • Use cumulative coverage (object-by-object) to compute minimal cache bytes for your hit-rate target.
  • Size NVMe-oF capacity for both capacity and the backend miss IOPS/throughput it implies.
  • Explicitly calculate flash DWPD based on your write profile and add operational headroom.
  • Validate with trace replays and gate-based acceptance tests; vendor benchmarks are useful but not a substitute for your workload testing.
