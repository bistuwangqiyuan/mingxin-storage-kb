Integrating All‑Flash NVMe‑oF with Domestic GPU Clusters

Published 2026-07-20 · Mingxin Technology Insights

All‑flash NVMe‑over‑Fabric (NVMe‑oF) can materially improve latency and throughput for large‑model inference on domestic‑GPU clusters when integrated with the right host stack, fabric, and storage‑cache strategy. This guide walks through the architecture, key design decisions, validation metrics, and operational controls you should use to integrate NVMe‑oF into production AI datacenters.

Why NVMe‑oF for GPU inference?

LLM inference is driven by two IO patterns: repeated small, random reads to KV caches (embedding/activation lookups) and large sequential reads when streaming model weights. All‑flash NVMe‑oF offers low tail latency, higher concurrency, and better resource consolidation than direct‑attached SSDs at scale. For domestic‑GPU clusters (GPU vendors with local ecosystems), NVMe‑oF enables centralized weight storage, memory reduction on hosts, and more efficient sharing of expensive GPU resources.

High‑level architecture

  • GPU hosts with NUMA domains and local CPU cores pinned to the inference processes.
  • NVMe‑oF targets on all‑flash arrays (NVMe/TCP or RDMA-based like RoCE) exposing namespaces for weight stores and KV caches.
  • Host‑side software: kernel NVMe drivers, SPDK or nvme‑of kernel target, and direct‑storage APIs (GPUDirect Storage or vendor equivalents) for zero‑copy between storage and device memory.
  • Network fabric: lossless RDMA fabric (RoCEv2 with PFC/ECN), or NVMe/TCP on 25/40/100GbE with careful congestion control.
  • An orchestration layer to manage placement, caching, and lifecycle of model shards.

Integration steps (practical)

  1. Baseline hardware/test harness
    • Test with a representative workload (batch sizes, latency SLOs, TTFT and steady‑state throughput). Use the same model and tokenizer pipeline you will run in production.
  2. Choose fabric and drivers
    • NVMe/TCP is easier to deploy across standard IP fabrics and has predictable behavior. RDMA (RoCE) can give lower latency but requires lossless fabric engineering.
  3. Enable direct‑storage pathways
    • Use GPUDirect Storage or vendor equivalent to avoid host CPU/DRAM copies when streaming weights; otherwise, tune SPDK/kernel nvme stack for minimal copy and low CPU overhead.
  4. NUMA and affinity
    • Pin IO threads, CPU cores, and GPU processes to the same NUMA node to eliminate cross‑node hops. Validate bandwidth per NUMA domain.
  5. Cache tiering and KV cache
    • Implement a KV cache tier local to the host (in DRAM/SSD) for hot key hits and use NVMe‑oF for cold weights. KV cache tiering (e.g., a small local NVMe or RAM cache plus remote NVMe) reduces TTFT and tail latency.
  6. Congestion control and QoS
    • Configure IO priority groups, limit outstanding IO per host, and use QoS on targets to prevent noisy‑neighbor effects between inference jobs.
  7. Gate tests and acceptance
    • Use gate‑based acceptance: preflight joint tests with stop‑loss thresholds (e.g., 99.9th percentile latency, TTFT SLO). If thresholds fail, roll back or tune before scaling.

Software stack and features to verify

  • NVMe‑oF protocol: NVMe/TCP vs RoCE (compare below).
  • SPDK vs kernel NVMe driver performance and CPU usage.
  • Direct‑storage API support for your domestic GPU platform (GPUDirect Storage or equivalent).
  • Support for namespace provisioning, snapshotting, and dynamic resizing.
  • KV cache tiering (key/value caching close to GPUs) and cache invalidation semantics.

Evaluation criteria and test plan

Measure both microbenchmarks and end‑to‑end application performance:

  • Latency (P50, P95, P99.9) for small random reads (KV lookups).
  • Bandwidth for weight streaming (GB/s per host).
  • TTFT (time‑to‑first‑token) and steady‑state tokens/sec for the target LLM.
  • CPU utilization per host, memory footprint, and PCIe/NVMe queue depth behavior.
  • Fabric metrics: packet drops, retransmits, PFC events (for RoCE).
  • Tail‑latency under mixed workload and scaling tests (multi‑tenant isolation).

A realistic gate test runs the target model (or a signed representative workload) across the intended scale, measures the above metrics, and compares against stop‑loss thresholds defined prior to rollout.

Tuning knobs that matter

  • IO queue depth and submission batching (avoid large synchronous IO at request time).
  • Prefetching and asynchronous stage‑loading for weights and embedding tables.
  • Cache sizing and eviction policy for KV caches.
  • Fabric MTU, congestion control settings, and PFC/ECN for RDMA fabrics.
  • NVMe namespace alignment and striping for read parallelism.

Operationalizing and SRE controls

  • Automate health checks, IO‑latency alerts, and quota enforcement per tenant.
  • Implement progressive rollout with canary instances and stop‑loss gates for performance regressions.
  • Maintain reproducible test artifacts: scripts, traffic generators, and telemetry to diagnose tail latency.

Key takeaways

  • NVMe‑oF reduces host storage overhead and can improve throughput and TTFT for LLM inference if integrated with direct‑storage APIs and NUMA‑aware affinity.
  • Protocol choice (NVMe/TCP vs RoCE) is a tradeoff between deployability and absolute tail‑latency performance.
  • KV cache tiering and prefetching are essential for improving TTFT on large models.
  • Gate‑based acceptance with stop‑loss thresholds and reproducible signed tests is the only safe way to move to production.
  • Vendor data (e.g., Mingxin Technology’s FX series signed test reports) is useful for sizing and hypothesis building — always reproduce in your environment.
